## 一、总体设计思路

本详细设计以《夯实多模态数据管理能力》技术规范书为纲，结合集团"1+23+M+N"数据集蓝图、集团元数据采集规范V1.36、高质量数据集目录元数据v1.52，分五大模块展开设计：

| 规范书章节 | 对应设计模块 |
|---|---|
| 新增高质量数据集元数据 | 模块一：高质量数据集元模型设计 |
| 高质量数据集上传集团 | 模块二：多模态数据集目录构建与集团上报 |
| 构建集团—省两级盘点闭环 | 模块三：数据资产盘点闭环设计 |
| 其他元数据管理优化 | 模块四：元数据管理优化 |
| 接口安全适配改造 | 模块五：接口安全适配与EOP接入 |

## 模块一：高质量数据集元模型设计

### 1.1 元模型整体结构

对应规范书"新增高质量数据集元数据类型"，结合集团元数据v1.52的56个字段，设计四组元数据分组：

```
高质量数据集元数据
├── 基础信息组（Basic Info）
├── 存储信息组（Storage Info）
├── 结构属性信息组（Structure Attr）
└── 生命周期组（Lifecycle）
```

### 1.3 元模型配置能力设计

元模型采用可配置方式，支持管理员通过元模型配置界面：
- 新增/修改字段定义（字段名、类型、必填规则、枚举值）
- 配置条件必填规则（如数据集类型=知识库时，知识库相关字段必填）
- 配置字段展示分组

### 1.4 对接多模态管理平台

**数据同步流程：**

```
多模态管理平台│
    ├─ 定时任务（每日/触发式）
    │       │
    │       ▼
    │  调用多模态平台API
    │       │
    │       ├─ 获取基础信息、存储信息、结构属性、生命周期
    │       │       └─ 写入 t_dataset_metadata（元数据信息表）
    │       │
    │       └─ 获取数据集质量信息
    │               └─ 写入 t_dataset_quality（质量数据表）
    │
    └─ 同步完成后触发写入多模态上传集团中间表
```

**核心数据表设计：**

```sql
-- 元数据信息表
t_dataset_metadata (
  dataset_id        VARCHAR(255) PK,  -- 数据集唯一标识
  dataset_name      VARCHAR(255),
  dataset_type      VARCHAR(255),     -- 原始数据/知识库/语料/提示词
  org_unit          VARCHAR(255),
  org_dept          VARCHAR(255),
  biz_scene         VARCHAR(255),
  biz_sub_scene     VARCHAR(255),
  -- ... 全部56个字段
  oper_type         VARCHAR(10),      -- 0新增/1修改/2删除
  oper_time         VARCHAR(20),
  sync_time         DATETIME          -- 同步时间
)

-- 质量数据表
t_dataset_quality (
  quality_id        BIGINT PK AUTO_INCREMENT,
  dataset_id        VARCHAR(255),     -- 关联元数据
  quality_dim       VARCHAR(100),     -- 质量维度
  quality_score     DECIMAL(5,2),
  quality_desc      VARCHAR(500),
  check_time        DATETIME
)
```

## 模块二：多模态数据集目录构建与集团上报

### 2.1 高质量数据集目录构建

#### 2.1.1 目录体系设计

依据"1+23+M+N"蓝图，目录体系设计如下：

```
企业级高质量数据集目录（1个）
├── 云网运营领域数据集目录
│   ├── 运行维护
│   ├── 云网安全
│   └── 应急服务
├── 客户服务领域数据集目录
│   ├── 服务管理
│   ├── 客户关系
│   └── 质量监督
├── 全渠道领域数据集目录
├── 市场领域数据集目录
├── 网信安领域数据集目录
├── 采供领域数据集目录
└── [其他M个业务领域]└── [N个业务细类]
        ├── 知识库数据集
        └── 语料数据集
```

目录节点数据结构：

```sql
t_dataset_catalog (
  catalog_id        VARCHAR(255) PK,
  catalog_name      VARCHAR(255),
  parent_id         VARCHAR(255),     -- 父节点，根节点为NULL
  catalog_level     INT,              -- 层级：1/2/3/4
  catalog_type      VARCHAR(50),      -- 领域/场景/细类
  sort_order        INT,
  create_time       DATETIME,
  status            VARCHAR(20)
)
```

#### 2.1.2 目录查询与展示

**查询条件：**
- 数据集名称（模糊）
- 数据集类型（原始数据/知识库/语料/提示词）
- 所属单位/部门
- 业务场景/子场景
- 是否入湖
- 状态标识
- 是否合规

**列表展示字段：**
数据集名称、数据集唯一标识、数据集类型、所属单位、业务场景、更新频率、状态、是否合规

**详情页展示：**
- Tab1：基础信息（全部基础信息字段）
- Tab2：质量信息（来自t_dataset_quality，展示各质量维度得分）
- Tab3：存储信息
- Tab4：生命周期

### 2.2 元数据查询新增高质量数据集类型

在现有元数据查询入口新增"高质量数据集"资产类型选项，查询结果支持查看详情（基础信息+质量信息），与目录查询详情页保持一致。

#### 2.3.2 集团唯一标识生成规则

高质量数据集集团唯一标识生成原子能力：

```
规则：[组织编码]-[部门缩写]-[业务场景缩写]-[DATASET]-[顺序编码]
示例：JT-KF-HWXJ-DATASET-0001
```

生成逻辑：
1. 取所属单位对应组织编码（参照集团附录2）
2. 取所属部门缩写（不超8位）
3. 取业务场景缩写（不超8位）
4. 固定段：DATASET
5. 顺序编码：4-8位数字，按单位+部门+场景维度自增

#### 2.3.3 稽核规则配置

多模态类稽核规则（可配置）：

| 规则编码 | 规则名称 | 规则描述 |
|---|---|---|
| MM-001 | 非空校验 | 必填字段不能为空 |
| MM-002 | 唯一标识格式校验 | 数据集唯一标识符合命名规范 |
| MM-003 | 数据集类型一致性 | 知识库类型时，知识库扩展字段不能为空 |
| MM-004 | 语料类型一致性 | 语料类型时，语料扩展字段不能为空 |
| MM-005 | 入湖一致性 | 入湖=是时，存储位置不能为空 |
| MM-006 | 共享一致性 | 内部共享要求非"不共享"时，样例数据不能为空 |
| MM-007 | 敏感数据一致性 | 是否敏感=是时，敏感数据信息不能为空 |
| MM-008 | 样例数据条数 | 样例数据不少于10条 |

#### 2.3.4 多模态元数据集团上报功能

**页面功能设计：**

```
多模态元数据集团上报
├── 非结构化数据上传集团
│   ├── 筛选条件：文件名称、所属系统、是否合规
│   ├── 列表：文件唯一标识、中文名、英文名、所属系统、是否合规、不合规原因
│   ├── 操作：单条修改、批量修改、上传集团
│   └── 稽核结果展示（多条拼接）
│
└── 高质量数据集上传集团
    ├── 筛选条件：数据集名称、数据集类型、所属单位、是否合规
    ├── 列表：数据集名称、唯一标识、类型、所属单位、是否合规、不合规原因
    ├── 操作：单条修改、批量修改、上传集团
    └── 稽核结果展示（多条拼接）
```

**上传集团流程：**

```
用户点击"上传集团"│
    ├─ 系统校验稽核规则
    │       ├─ 合规 → 写入集团上传表（oper_type=0新增 or 1修改）
    │       └─ 不合规 → 展示不合规原因，阻断上传（可配置是否强制上传）
    │
    └─ 写入成功 → 更新upload_status=已上传，记录upload_time
```

## 模块三：数据资产盘点闭环设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│                  省级资产管理系统                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ 资源类    │  │ 资产类    │  │   多模态类        │  │
│  │ 中间表    │  │ 中间表    │  │   中间表          │  │
│  └────┬─────┘  └────┬─────┘  └────────┬─────────┘  │
│       │              │
│       └──────────────┴──────────────────┘            │
│                       │                              │
│              稽核规则引擎（可配置）                    │
│                       │                              │
│              集团上传表（9类资产）                     │
└───────────────────────┬─────────────────────────────┘
                        │ 增量上传（每月25日18:00前）
                        ▼
              集团数据资产管理平台
                        │
                        ▼稽核结果回写（dx_assets_*_result_*）
```

### 3.2 资源类数据处理流程

#### 3.2.1 中间表初始化

一次性初始化逻辑（伪代码）：

```
初始化资源类上传集团中间表：
  SELECT 
    s.*, db.*, t.*, f.*,
    -- 关联临时处理中间表补全缺失字段
    tmp.补全字段
  FROM 资产元数据库(系统/库/表/字段)
  JOIN 临时处理中间表 ON 关联条件
  JOIN 集团表结构映射 ON 字段映射
  → INSERT INTO t_upload_resource_system/database/table/field
```

#### 3.2.2 增量更新触发机制

```
安运中心推送数据库/表/字段元数据
    │
    ▼
同步到分级分类中间表
    │
    ▼
触发增量更新任务：
  - 对比上次同步时间，提取变更记录
  - 关联本地资产元数据（补全中文名、主题域等）
  - 更新到资源类上传集团中间表
  - 设置 oper_type=0(新增) or 1(修改)，oper_time=当前时间
```

#### 3.2.3 稽核规则引擎

稽核规则可配置，支持以下规则类型：

| 规则类型 | 规则说明 | 适用对象 |
|---|---|---|
| 非空规则 | 必填字段不能为空 | 全部 |
| 集团唯一标识生成规则 | 标识符合命名规范且唯一 | 全部 |
| 字段没在表中 | 字段的TBL_ID在表资产中不存在 | 字段 |
| 表没有字段 | 表下无任何字段记录 | 表 |
| 数据库没有表 | 数据库下无任何表记录 | 数据库 |
| 表没在数据库中 | 表的DBS_ID在数据库资产中不存在 | 表 |
| 系统没有数据库 | 系统下无任何数据库记录 | 系统 |
| 数据库没在系统中 | 数据库的SYS_ID在系统资产中不存在 | 数据库 |
| 数据库下有相同的表 | 同一数据库下表英文名重复 | 表 |
| 表下字段英文名重复 | 同一表下字段英文名重复 | 字段 |
| 样例数据小于10条 | 样例数据JSON解析后条数<10 | 表/指标/标签 |
| 样例数据全部一致 | 样例数据所有记录完全相同 | 表/指标/标签 |
| 对象属性数量和样例数据不一致 | 样例数据字段数与字段元数据数量不符 | 表 |
| 主题域取值范围校验 | 主题域值在允许范围内 | 表 |

稽核结果写入稽核结果表（dx_assets_*_result_*），多条不合规原因用"；"拼接后统一展示。

#### 6.2.3 改造要求

```
每个接口调用改造要点：
  1. 接入EOP统一网关地址（替换原直连地址）
  2. 请求头增加EOP鉴权Token
  3. 按EOP规范转换请求/响应报文
  4. 增加重试机制（3次，指数退避）
  5. 增加熔断保护（Hystrix或Sentinel）
  6. 统一日志记录（request_id贯穿）
```

                                      Object request, 
                                      Class<T> responseClass) {
        // 1. 构造EOP请求头（鉴权）
        // 2. 签名
        // 3. 调用EOP网关
        // 4. 统一异常处理、日志、重试
    }
