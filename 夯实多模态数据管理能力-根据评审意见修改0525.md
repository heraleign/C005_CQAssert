## 一、核心流程重新定义

根据评审意见，整体上报流程调整如下：

```
┌─────────────────────────────────────────────────────────────────┐
│                        完整上报流程                              │
││
│  本地元数据表                                    │
│  （自动补全在本地完成，走独立流程）                               │
│       │                                                          │
│       │ ① 手动触发同步（资产管理员按系统操作）                   │
│       │   - 全量同步到中间表                                     │
│       │   - 空值更新非空时提示确认                               │

│       ▼                                                          │
│  ┌─────────────────┐                                            │

│           ▼                                                      │

│           │ ③ 确认合规内容，点击同步                │
│           ▼                                                      │

│  │ 集团上传中间结果表   │  ← 只保留合规且可上传的数据            │
│  └────────┬────────────┘                                        │
│           │ ④ 确认后点击上传                │
│           ▼                                                      │
│  ┌─────────────────┐                                            │

│  └─────────────────┘                                            │
└─────────────────────────────────────────────────────────────────┘
```

## 二、数据表设计调整

### 2.1 集团上传中间表（核心调整）

**关键设计原则：**
- 唯一标识一旦生成不可重复生成，所有更新走 UPSERT
- 业务唯一标识（本地）与集团唯一标识分开存储
- 记录中间表修改日志

```sql
-- 以资源类-表元数据为例，其他类型同理
t_upload_mid_table (
  id                    BIGINT PK AUTO_INCREMENT,
  -- 本地业务唯一标识（UPSERT的KEY）
  local_biz_id          VARCHAR(255) NOT NULL,  -- 本地表唯一标识

  -- 集团唯一标识（一旦生成不可变更）
  group_tbl_id          VARCHAR(255),-- 集团表唯一标识 TBL_ID
  group_id_generated    TINYINT DEFAULT 0,      -- 0=未生成 1=已生成

  -- 集团规范字段（对应dx_assets_table全部字段）
  table_schema          VARCHAR(255),
  table_en_name         VARCHAR(255),
  table_name            VARCHAR(255),
  table_introduct       VARCHAR(500),
  dbs_id                VARCHAR(255),           -- 关联集团数据库唯一标识
  table_domain          VARCHAR(255),
  lake_data_type        VARCHAR(10),
  premium_model_in_lake VARCHAR(10),
  uploaded_to_big_lake  VARCHAR(10),
  is_shared             VARCHAR(10),
  sample_data           LONGTEXT,
  create_time_col       VARCHAR(20),
  is_partitioned        VARCHAR(10),
  oper_type             VARCHAR(10),-- 0新增/1修改/2删除
  oper_time             VARCHAR(20),
  -- ... 其他集团规范字段

  -- 中间表管理字段
  sync_source           VARCHAR(50),            -- 同步来源：manual（手动）
  last_sync_time        DATETIME,               -- 最后同步时间
  mid_modify_flag       TINYINT DEFAULT 0,      -- 0=未修改 1=已在中间表修改

  -- 稽核字段
  audit_status          VARCHAR(20),            -- pending/pass/fail
  audit_timeDATETIME,
  non_compliant_reason  VARCHAR(2000),          -- 多条拼接

  -- 上传状态
  upload_status         VARCHAR(20),            -- pending/synced/uploaded/failed

  -- 盘点相关
  if_managed            VARCHAR(10),            -- 是否需要纳管/盘点（来自系统元数据）
  sys_status            VARCHAR(10),            -- 系统状态（建设中/在用/已下线）
  sys_func_type         VARCHAR(10),            -- 系统功能类型（1纯数据/2纯功能/3数据+功能）
  UNIQUE KEY uk_local_biz_id (local_biz_id)
)
```

**UPSERT 逻辑（关键）：**

```sql
INSERT INTO t_upload_mid_table (local_biz_id, table_en_name, table_name, ...)
VALUES (#{localBizId}, #{tableEnName}, #{tableName}, ...)
ON DUPLICATE KEY UPDATE
  table_en_name    = VALUES(table_en_name),
  table_name       = VALUES(table_name),
  -- 空值更新非空保护：只有源值非空时才更新
  table_introduct  = IF(VALUES(table_introduct) IS NOT NULL 
                        AND VALUES(table_introduct) != '', 
                        VALUES(table_introduct), table_introduct),
  -- 集团唯一标识：已生成则不覆盖
  group_tbl_id     = IF(group_id_generated = 1, group_tbl_id, VALUES(group_tbl_id)),
  last_sync_time   = NOW(),
  oper_type        = VALUES(oper_type),
  oper_time        = VALUES(oper_time)
```

### 2.2 中间表字段自动生成规则配置

根据集团规范，部分字段可由其他字段推导生成，设计规则配置表：

```sql
t_mid_field_gen_rule (
  rule_id           BIGINT PK AUTO_INCREMENT,
  asset_type        VARCHAR(50),    -- system/database/table/field/...
  target_field      VARCHAR(100),   -- 目标字段（待生成）
  rule_type         VARCHAR(50),    -- COPY/CONCAT/ENUM_MAP/FORMULA
  source_fields     VARCHAR(500),   -- 来源字段（JSON数组）
  rule_expr         VARCHAR(1000),  -- 规则表达式
  trigger_condition VARCHAR(500),   -- 触发条件（如：target_field IS NULL）
  is_enabled        TINYINT DEFAULT 1,
  remark            VARCHAR(500)
)
```

**典型规则示例（对应集团规范要求）：**

| 目标字段 | 规则类型 | 规则说明 |
|---|---|---|
| TBL_ID（集团表唯一标识） | FORMULA | 当group_id_generated=0时，按[组织编码]-[部门]-[系统]-TABLE-[序号]生成 |
| OPER_TIME | COPY | 取当前时间 YYYYMMDDHHMMSS |
| UPLOADED_TO_BIG_LAKE | ENUM_MAP | 湖外系统统一填0 |
| PREMIUM_MODEL_IN_LAKE | ENUM_MAP | 湖外系统统一填0 |
| IS_SHARED | ENUM_MAP | 湖外系统统一填0 |
| DBS_ID | COPY | 从本地数据库元数据映射到集团数据库唯一标识 |

### 2.3 本地元数据到中间表字段映射配置

```sql
t_metadata_field_mapping (
  mapping_id        BIGINT PK AUTO_INCREMENT,

  local_field       VARCHAR(100),     -- 本地字段名

  mid_field         VARCHAR(100),     -- 中间表字段名
  transform_rule    VARCHAR(500),     -- 转换规则（为空则直接映射）
  is_enabled        TINYINT DEFAULT 1

```

**映射示例（资源类-表）：**

| 本地字段 | 中间表字段 | 转换规则 |
|---|---|---|
| table_en_name | TABLE_EN_NAME | 直接映射 |
| table_cn_name | TABLE_NAME | 直接映射 |
| table_desc | TABLE_INTRODUCT | 直接映射 |
| classify_level | TABLE_LEVEL | 枚举值转换 |
| is_lake | LAKE_DATA_TYPE | 是→0，否→1 |
| theme_domain | TABLE_DOMAIN | 直接映射 |

## 三、页面功能详细设计

### 3.1 资源类集团上报主页面

#### 3.1.1 页面整体布局

```
┌─────────────────────────────────────────────────────────────┐
│  资源类元数据集团上报                │
├─────────────────────────────────────────────────────────────┤

│  [操作区]                                                     │

```

#### 3.1.2 筛选逻辑

**建设中不盘、上线后纯功能不盘** 的筛选规则：

```
默认过滤条件（可手动取消）：
  - 排除 sys_status = '建设中'
  - 排除 sys_func_type = '2'（纯功能型）且 sys_status = '在用'

页面提供筛选项：
  - 系统状态：全部 / 建设中 / 在用 / 已下线
  - 系统功能类型：全部 / 纯数据型 / 纯功能型 / 数据+功能型
  - 是否需要盘点：全部 / 是 / 否
```

#### 3.1.3 系统与库表分别上传

**分层上传设计：**

```
系统Tab：
  - 每行系统记录独立操作
  - 操作按钮：[仅上传系统] [上传系统+库表]
  - 状态字段：系统上传状态、库表上传状态（分开显示）

说明文字：
  "系统建设中或库表未完成盘点时，可先仅上传系统元数据。当月25日前，盘点范围内的库表必须完成上传。"

强制提示逻辑：
  IF 当前日期 >= 本月20日 
  AND 系统.if_managed = '1'（需要盘点）
  AND 库表上传状态 = '未上传'：页面显示红色警告："距本月25日截止还有X天，该系统库表尚未上传！"
```

### 3.2 同步到中间表功能

#### 3.2.1 触发方式

```
入口1：系统Tab → 选择系统 → [同步到中间表]- 按系统粒度同步：该系统下的系统/库/表/字段全量同步

入口2：各Tab → 选择记录 → [同步到中间表]
  - 按类型粒度同步：仅同步选中类型的记录

入口3：操作区 → [同步到中间表 ▼] → 全量同步
  - 全量同步所有符合筛选条件的记录
```

#### 3.2.2 空值更新非空提示

```
同步执行前检查：
  FOR EACH 待同步记录:FOR EACH 字段:
          IF 中间表当前值 != NULL AND 中间表当前值 != ''
          AND 本地元数据源值 == NULL OR 源值 == '':
              加入"空值覆盖警告列表"

IF 警告列表不为空:
    弹出确认框：
    ┌─────────────────────────────────────────┐
    │  以下字段将被空值覆盖，请确认：│
    │  系统名称 | 字段名 | 当前值 | 新值(空)   │
    │  XXX系统  | 表中文名 | 客户信息表 | (空) │
    │  ...│
    │  [取消] [跳过空值更新] [确认全部覆盖]    │
    └─────────────────────────────────────────┘
  
    - 取消：不执行同步
    - 跳过空值更新：执行同步，但空值字段不覆盖非空值
    - 确认全部覆盖：执行同步，空值也覆盖
```

#### 3.2.3 中间表字段自动生成

同步完成后，自动触发字段生成规则：

```
FOR EACH 同步记录:
    FOR EACH 启用的生成规则 WHERE asset_type匹配:
        IF 触发条件满足（如目标字段为空）:
            按规则计算目标字段值
            UPDATE 中间表
特殊规则：集团唯一标识生成
    IF group_id_generated = 0:
        生成集团唯一标识
        SET group_id_generated = 1
        -- 一旦生成，后续同步不再覆盖
```

### 3.3 稽核功能

#### 3.3.1 触发稽核

```
入口：
  - 单条：操作列 [稽核]
  - 批量：选择多条 → [触发稽核]
  - 按系统：选择系统 → [触发稽核▼] → 稽核该系统下全部
  - 全量：[触发稽核▼] → 全量稽核

稽核执行：
  1. 读取 t_audit_rule（启用的规则）
  2. 逐条执行规则校验
  3. 多条不合规原因用"；"拼接
  4. 更新中间表 audit_status / non_compliant_reason / audit_time
  5. 写入操作日志
```

#### 3.3.2 稽核结果查看

```
稽核结果展示列：
  稽核状态（通过/不通过/待稽核）| 不合规原因 | 稽核时间

筛选：稽核状态 = 不通过 → 快速定位问题记录

不合规原因展示：
  - 列表中截断显示（超过50字符省略）
  - 鼠标悬停/点击展开全部原因
  - 多条原因以列表形式展示
```

### 3.4 中间表修改功能

**核心约束：只能修改中间表，不能修改本地元数据表**

#### 3.4.1 修改入口

```
入口：稽核不通过的记录 → 操作列 [修改中间表]

弹出修改表单：
  - 展示当前中间表字段值
  - 标红显示不合规字段
  - 不合规原因提示显示在对应字段旁
  - 可编辑字段（集团规范允许填写的字段）
  - 不可编辑：集团唯一标识（已生成）、本地业务唯一标识
底部按钮：[取消] [保存] [保存并重新稽核]
```

#### 3.4.2 修改日志记录

```
每次保存：
  - 记录修改前后值到 t_mid_field_modify_log
  - 更新中间表 mid_modify_flag = 1
  - 重置 audit_status = 'pending'（需重新稽核）
```

#### 3.4.3 修改后再次稽核

```
[保存并重新稽核] 按钮：
  - 保存修改
  - 立即对该条记录执行稽核
  - 刷新稽核状态显示
```

### 3.5 同步到中间结果表

```
触发条件：稽核状态 = pass

操作：
  - 单条：[同步到结果表]
  - 批量：选择多条通过记录 → [同步到结果表]
  - 全量：[同步到结果表] → 同步所有稽核通过记录

执行逻辑：
  1. 过滤 audit_status = 'pass' 的记录
  2. 生成数据快照（JSON格式，集团规范全字段）
  3. UPSERT 到 t_upload_result_table
  4. 更新中间表 upload_status = 'synced'
  5. 写入操作日志

确认弹框：
  "将同步 X 条合规记录到上传结果表，确认？"
  [取消] [确认]
```

### 3.6 上传集团功能

```
触发条件：upload_status = 'synced'（已同步到结果表）

上传模式：
  ┌─ 按系统上传：选择系统 → 上传该系统下全部已同步记录
  ├─ 按类型上传：选择Tab（系统/库/表/字段）→ 上传该类型
  └─ 全量上传：上传所有已同步记录

执行逻辑：
  1. 读取 t_upload_result_table（upload_status = 'pending'）
  2. 写入集团结果表（dx_assets_*_[单位缩写]）
  3. 更新 upload_status = 'uploaded'，upload_time = NOW()
  4. 生成 upload_batch_no（批次号：YYYYMMDDHHMMSS+随机4位）
  5. 写入操作日志

上传结果提示：
  "本次上传：成功 X 条，失败 Y 条"
  失败记录可查看失败原因，支持重试
```

### 3.7 操作记录查询页面

#### 3.7.1 修改记录查询

```
查询条件：
  资产类型 | 系统名称 | 操作人 | 修改时间范围 | 字段名

列表展示：
  操作时间 | 操作人 | 资产类型 | 记录标识 | 字段名 | 修改前值 | 修改后值 | 修改原因
```

#### 3.7.2 上传记录查询

```
查询条件：
  资产类型 | 系统名称 | 上传批次号 | 上传时间范围 | 上传状态

列表展示：
  批次号 | 上传时间 | 操作人 | 资产类型 | 上传范围 | 成功条数 | 失败条数 | 状态

支持：
  - 点击批次号查看该批次明细
  - 失败记录支持重新上传
```

## 四、资产类上报流程设计

### 4.1 资产类（指标/标签/API/产品）同步流程

```
按类型同步（与资源类流程一致，但按资产类型分开操作）：

指标Tab：
  [同步到中间表] → 从本地指标元数据表同步
  [触发稽核] → 执行指标类稽核规则
  [同步到结果表] → 合规指标写入结果表
  [上传集团] → 写入集团指标结果表

标签/API/产品 同理
```

### 4.2 资产类中间表设计（以指标为例）

```sql
t_upload_mid_indicator (
  idBIGINT PK AUTO_INCREMENT,
  local_biz_id      VARCHAR(255) NOT NULL,  -- 本地指标唯一标识（UPSERT KEY）
  group_ind_id      VARCHAR(255),           -- 集团指标唯一标识（生成后不变）
  group_id_generated TINYINT DEFAULT 0,

  -- 集团规范字段（对应dx_assets_indicator）
  first_level_category  VARCHAR(255),
  second_level_category VARCHAR(255),
  third_level_category  VARCHAR(255),
  fourth_level_category VARCHAR(255),
  ind_name              VARCHAR(255),
```

## 五、本地元数据补全能力扩展

### 5.1 现有补全能力问题

规范书及评审意见指出：**本地自动元数据补全功能缺失按系统进行补全的功能**。

### 5.2 补全能力扩展设计

在本地元数据管理模块（独立流程）新增按系统/库/表/字段粒度的批量补全入口：

```
补全入口层级：
  ├─ 按系统补全：选择系统 → 补全该系统下所有缺失字段
  │     补全范围：系统下所有表的中文名、简介、主题域
  │系统下所有字段的中文名
  │
  ├─ 按库补全：选择数据库 → 补全该库下所有表/字段
  │
  ├─ 按表补全：选择表 → 补全该表的中文名、简介、主题域
  │               + 该表下所有字段中文名
  │
  └─ 按字段补全：选择字段 → 补全字段中文名
```

**补全任务设计：**

```sql
t_metadata_complete_task (
  task_id           VARCHAR(50) PK,   -- 任务ID
  task_type         VARCHAR(50),      -- SYSTEM/DATABASE/TABLE/FIELD
  scope_id          VARCHAR(255),     -- 补全范围ID
  scope_name        VARCHAR(255),
  complete_fields   VARCHAR(500),     -- 待补全字段列表（JSON）
  total_count       INT,              -- 总条数
  success_count     INT DEFAULT 0,
  fail_count        INT DEFAULT 0,
  task_status       VARCHAR(20),      -- pending/running/done/failed
  operator          VARCHAR(100),
  create_time       DATETIME,
  finish_time       DATETIME
)
```

**补全执行逻辑：**

```
1. 创建补全任务，返回task_id
2. 异步执行（避免大批量阻塞页面）
3. 按批次调用AI补全原子能力
4. 更新本地元数据表（只更新空值字段，不覆盖已有值）
5. 更新任务进度
6. 完成后通知操作人（页面提示/消息）

注意：补全完成后，需手动触发同步到中间表，补全结果不自动同步（保持流程独立）
```

## 六、关键设计约束汇总

| 约束 | 设计实现 |
|---|---|
| 集团唯一标识不可重复生成 | group_id_generated标志位，UPSERT时已生成则不覆盖 |
| 不能清空后全量插入 | 全部使用 ON DUPLICATE KEY UPDATE（按local_biz_id） |
| 不能修改本地元数据表 | 中间表修改入口明确标注，后端接口只更新中间表 |
| 空值更新非空需提示 | 同步前检查，弹框三选一确认 |
| 建设中/纯功能不盘 | 页面默认过滤，可手动调整筛选条件 |
| 系统与库表可分别上传 | 上传操作分层，系统Tab提供"仅上传系统"选项 |
| 修改中间表需记录日志 | t_mid_field_modify_log，字段级别记录 |
| 稽核不通过可修改后再稽核 | [保存并重新稽核]按钮，修改后重置audit_status=pending |
| 自动补全不在上报流程处理 | 补全在本地元数据独立流程，上报流程只做同步 |
| 中间表字段可按规则自动生成 | t_mid_field_gen_rule配置表，同步后自动触发 |
| 本地到中间表需做字段映射 | t_metadata_field_mapping配置表，支持转换规则 |
| 合规内容才进结果表 | 同步到结果表时过滤audit_status=pass |
| 上传记录可查询 | t_upload_operation_log，支持批次查询和明细查询 |

## 十、25日截止日期管控机制

### 10.1 截止日期预警

```
定时任务：每日08:00执行

  计算距本月25日剩余天数 = 25 - 当前日期

  IF 剩余天数 <= 5:
    查询所有 if_managed=1 AND sys_status=在用 AND sys_func_type!=2 的系统
    统计各系统库表上传状态
    FOR EACH 系统 WHERE 库表upload_status != 'uploaded':
        生成预警记录
        推送消息给数据资产管理员

  IF 当前日期 > 25:
    标记本月截止日期已过，生成未完成上传报告
```

### 10.2 页面截止日期提示

```
页面顶部Banner（距25日<=5天时显示）：

└─────────────────────────────────────────────────────────┘

系统列表中，未完成上传且需要盘点的系统行：
  - 行背景色标红（距截止<=3天）或标黄（距截止<=5天）
  - 库表上传状态列显示："⚠️ 未上传（距截止X天）"
```

**至此，根据评审意见的详细设计调整已全部完成。**

核心变更总结：
1. **流程重构**：明确五步流程（本地同步→中间表稽核→修改→结果表→上传集团），各步骤均有手动触发入口
2. **UPSERT机制**：所有中间表按业务唯一标识做UPSERT，集团唯一标识一旦生成不可覆盖
3. **修改隔离**：只允许修改中间表，本地元数据表不可在此流程修改，修改全程记录日志
4. **补全解耦**：自动补全从上报流程中移除，在本地元数据独立流程处理，并扩展按系统/库/表/字段粒度补全能力
5. **分层上传**：系统与库表可独立上传，25日截止日期有预警机制
